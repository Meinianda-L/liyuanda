FileCairn: Local File Intelligence

A practical local-first tool that scans, indexes, and ranks personal files so users can retrieve what they need quickly through CLI or API.

Written in Python by Kevin, then translated to Go.

Overview

FileCairn is a local search system written in Go with two interfaces: CLI and HTTP API. It scans selected roots, stores normalized file records, builds an inverted index (token -> file IDs), and ranks candidates by weighted relevance. The design goal is fast retrieval from context words, not exact filename memory.

Why This Is Useful

Faster daily search

Users can search with phrases like "math worksheet draft" and still find files when the exact filename is forgotten.

Privacy-first local model

Everything runs on-device. Content indexing is disabled by default and must be explicitly enabled.

Tool integration ready

Local endpoints (/v1/find, /v1/scan, /v1/register-created) support linked tools and automation flows.

Core Workflow (Ordered Spec)

  1. Initialize configuration with filecairn init (roots, ignore list, content-read mode, provider metadata).
  2. Scan roots with filecairn scan; directory walker skips ignored folders and non-regular files.
  3. Reuse unchanged files by matching (size, mod_time); only changed files are re-tokenized.
  4. Persist records to ~/.filecairn/files.json and rebuild postings map in ~/.filecairn/inverted.json.
  5. Run filecairn find \"query\"; query tokens pull candidates from inverted index, then scoring/ranking executes.
  6. Expose same behavior to external tools through filecairn serve and local HTTP endpoints.
  7. Whenever another tool writes a file, call register-created so it becomes searchable immediately.

Concrete Command Sequence

filecairn init
filecairn scan --roots ~/Documents,~/Desktop
filecairn find "meeting notes draft"
filecairn serve --addr 127.0.0.1:4819
filecairn register-created --path /tmp/new-note.md --agent openclaw --share full

Algorithm Details (LaTeX)

Mathematical form of the ranking pipeline:

1) Candidate retrieval from inverted index
\[ \mathcal{C}(q)=\bigcup_{\tau \in \mathcal{T}(q)} I(\tau) \]
2) Token match component
\[ T(d,q)=\sum_{\tau \in \mathcal{T}(q)} \mathrm{tf}(\tau,d) \]
3) Recency decay component
\[ R(d)=e^{-\mathrm{ageDays}(d)/45}, \qquad \mathrm{ageDays}(d)=\frac{t_{\mathrm{now}}-t_{\mathrm{mod}}(d)}{86400} \]
4) Final weighted ranking score
\[ \mathrm{Score}(d,q)=0.45\,T(d,q)+0.30\,R(d)+0.15\,P(d)+0.10\,L(d,q) \]
5) Incremental scan reuse rule
\[ \mathrm{Reuse}(d)= \big[\mathrm{size}(d)=\mathrm{size}_{\mathrm{old}}(d)\big] \land \big[t_{\mathrm{mod}}(d)=t_{\mathrm{mod,old}}(d)\big] \]
  1. Tokenize query and files: regex extraction plus stop-word filtering.
  2. Fetch candidates quickly: union of postings lists from the inverted index.
  3. Compute weighted score: token match, recency, path boost, and label boost.
  4. Sort results: descending by score, then newer modified time for tie-breaks.
  5. Return top N: path, score, labels, snippet (if available), and why-matched explanation.

Data Files and API Contract

Data directory: ~/.filecairn

  1. config.json - runtime settings and scan roots
  2. files.json - indexed file records
  3. inverted.json - token-to-file postings map
  4. agent_created.jsonl - append-only log of tool-created files

API endpoints: GET /v1/health, GET /v1/status, POST /v1/find, POST /v1/scan, POST /v1/register-created

Skills Integration

FileCairn ships skill files so linked tools can query the same local index and register created files automatically.

Start local API first:

/Users/kevin/filecairn/bin/filecairn serve --addr 127.0.0.1:4819

Install skill files:

mkdir -p "$HOME/.filecairn/skills"
cp /Users/kevin/filecairn/skills/*.md "$HOME/.filecairn/skills/"

Request Lifecycle Diagram

Bottom summary: how FileCairn handles scan requests and search requests.

Scan Pipeline

1. Read scan roots from config or request
->
2. Walk folders and skip ignored directories
->
3. Reuse unchanged files by size + mod time
->
4. Tokenize, label, and optional snippet extraction
->
5. Save files.json and rebuild inverted.json

Query Pipeline

1. Accept query and normalize limit
->
2. Tokenize query and fetch candidate file IDs
->
3. Compute weighted score per candidate
->
4. Sort by score, then by recency for ties
->
5. Return top results with why-matched info
